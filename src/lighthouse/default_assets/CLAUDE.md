# WorkAgent Wiki

This is my living knowledge base about the people and projects in my life. An LLM agent maintains it by watching signals from my digital life (email, calendar, messages, screenshots, chats with it). I read and edit it in Obsidian. This file is the shared agreement between me and the agent about how the wiki should feel.

If you're the agent reading this: treat this document as the source of truth for how to write. When it conflicts with your built-in prompt, this wins.

## What lives here

Two folders, nothing else:

- `people/` — one page per real person I actually interact with. Family, friends, coworkers, collaborators. Not celebrities, not strangers cc'd on a thread, not mailing lists.
- `projects/` — everything ongoing I care about. This is a deliberately broad category: real projects (a codebase, a launch, a trip, a move), goals I'm working toward, life threads I'm navigating (a kid's sports season, a house renovation), situations I'm managing (a health protocol, a hiring decision). If it has momentum and I care about where it goes, it's a project. Don't overthink the classification — the point of a single category is that I don't have to.

People link to the projects they're part of. Projects link to the people involved and to related projects. Use `[[wiki links]]` freely — the graph is half the point.

## How pages should read

Short prose. 100 to 400 words. A page should read like a smart friend briefing me on the current state of something — not a changelog, not a resume, not a CRM record.

- Write in flowing sentences, not bullet walls.
- Lead with what's true right now. History only if it matters to the present.
- Be concrete. "Shipped the notch UI last week" beats "made progress on frontend."
- Link people and projects with `[[brackets]]` on first mention.
- No dated log sections. No "Updates 2026-04-04:" headers. Edit in place.
- No tables of metadata unless the page genuinely needs one.

If a page is getting long, that's a signal something should split off into its own page, or that old material should be cut.

## The prime directive

Only write what the signals actually tell you.

This is the rule that matters most. The wiki is valuable because it's true. A wiki full of plausible-sounding inferences is worse than an empty one, because I'll start trusting it and get burned.

- Don't infer by analogy. If someone is a software engineer, don't guess they "probably use GitHub."
- Don't fill gaps with generic filler. Empty is fine. Short is fine.
- If something is uncertain, say so in the prose ("seems to be," "mentioned once in passing").
- If a signal contradicts what's on the page, update the page — don't stack the new claim on top of the old one.
- **Doing nothing is a valid outcome.** If today's signals don't meaningfully change any page, don't edit anything. No busywork.

## What this wiki is not

- Not a to-do list. No action items, no "next steps," no checkboxes. I have other tools for that.
- Not a suggestion engine. Don't propose new goals for me. Don't recommend I reach out to someone. Just describe what's there.
- Not an inbox digest — but also not a blind filter. Don't auto-dismiss anything by source type. A marketing email might be a flight confirmation; a receipt might reveal a new subscription; a one-off page view might be the start of something real. Reason about each signal on its own merits: does it actually change what you know about one of my people or projects? If yes, file it. If no, skip it. The bar is significance to me, not category of source.
- Not a diary. Don't narrate my days. Narrate the state of things.

## How edits work

Merge in place. When new information arrives, rewrite the relevant sentences so the page still reads as one coherent paragraph about the current state. Remove claims that are now stale or contradicted. Keep the prose tight.

Think of it like updating a Wikipedia article, not appending to a log file.
